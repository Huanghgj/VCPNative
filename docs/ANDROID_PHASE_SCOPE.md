# Android 分阶段范围

更新时间：2026-03-25

本文回答两个问题：

1. Android 首版到底要做到什么程度
2. 哪些功能必须明确后置，避免范围失控

---

## 1. 总体原则

Android 端当前不是做“全功能桌面复刻”，而是做“兼容现有 VCPToolBox 的 VCPChat 原生客户端”。

排序原则只有三条：

1. 先保证主链路可用
2. 先保证兼容性，再追求炫功能
3. 优先迁客户端必需能力，后迁桌面附属能力

---

## 2. P0：文档冻结

目标：

- 写清楚边界
- 写清楚本地数据
- 写清楚网络与流式契约
- 写清楚哪些功能进首版，哪些不进

完成条件：

- 已有高层方案文档
- 已有迁移映射文档
- 已有 `AppData` 规格
- 已有网络/流式契约规格
- 已有 Android 分阶段范围文档

当前阶段就在 P0。

---

## 3. P1：单聊 MVP

### 3.1 P1 必须包含

- 全局设置页
- `vcpServerUrl` / `vcpApiKey` 配置
- Agent 列表
- Topic 列表
- 打开最后一次聊天对象
- 单聊 `history` 读取
- 单聊 `history` 保存
- 文本发送
- SSE 流式回复
- 本地中断 + 远程中断
- 基础图片附件
- 基础消息气泡渲染
- 基础错误态

### 3.2 P1 可以先简化

- 只支持最常见消息样式
- 只支持最基础 Markdown
- 先不做复杂 HTML/DIV/Canvas 气泡
- 先不做富动画
- 先不做 Notes / Forum / Memo / Music / Voice Chat

### 3.3 P1 明确不包含

- 群聊
- 群邀请发言
- redo group message
- VCPLog
- context sanitizer
- context folding
- 正则规则
- 文档附件文本抽取
- Desktop Push 相关行为
- 桌面监听器

### 3.4 P1 验收标准

P1 至少要满足：

1. 用户能选 Agent 和 Topic
2. 用户能发送文本到现有 `VCPToolBox`
3. Android 能稳定收流
4. 用户能中断回复
5. 重启后还能读回本地历史

只要这五条不稳，就不应继续扩范围。

---

## 4. P2：兼容性补齐

### 4.1 P2 目标

让 Android 端从“能聊”提升到“和桌面端主要行为一致”。

### 4.2 P2 应补齐

- `enableVcpToolInjection`
- 模型参数透传
- `top_p` / `top_k` / `contextTokenLimit`
- 正则规则
- 思维链剥离
- context sanitizer
- context folding
- txt 附件文本抽取
- pdf 附件文本抽取
- docx 附件文本抽取
- 图片、音频、视频多模态兼容
- VCPLog

### 4.3 P2 仍可后置

- 群聊
- Notes
- Forum
- Memo
- Music
- Canvas 深度集成
- Voice Chat

---

## 5. P3：群聊迁移

### 5.1 P3 必须包含

- Group 列表
- Group 配置读取
- Group Topic 读写
- 群聊历史读写
- `sequential`
- `naturerandom`
- `invite_only`
- 邀请某个 Agent 发言
- redo 某条群消息
- 群聊流式状态机

### 5.2 P3 风险最高的点

- 群聊上下文由客户端本地编排
- 群聊请求字段与单聊不统一
- 群聊事件模型比单聊复杂
- 群聊 topic 总结也依赖客户端逻辑

结论：

- 群聊必须单独作为一个阶段
- 不能混进 P1

---

## 6. P4：高级模块筛选迁移

### 6.1 候选模块

- Notes
- Forum
- Memo
- Music
- Canvas
- Voice Chat
- Assistant Listener

### 6.2 进入 P4 的前提

只有在下面三件事稳定后，才允许进入 P4：

1. 单聊主链路稳定
2. 兼容性增强完成
3. 群聊主链路稳定

### 6.3 P4 的判断标准

每个高级模块都要单独判断：

- 是否真的需要移动端
- 是否依赖桌面宿主能力
- 是否会拖垮首版复杂度
- 是否与当前 Android 交互形态匹配

不能默认“桌面有的，移动端都要有”。

---

## 7. 明确排除项

下面这些内容即使以后要做，也不应该作为 Android 首版阻塞项：

- Electron 窗口体系
- 系统托盘
- 桌面悬浮窗
- 全局剪贴板监听策略照搬
- 全局键盘监听
- 桌面路径探测与注入行为照搬
- VCPHumanToolBox
- 大量桌面 HTML/DIV/Canvas 宿主玩法
- 任何“先全搬再裁”的迁移方式

---

## 8. 推荐的开发排序

推荐顺序如下：

1. 文档冻结
2. 单聊数据层
3. 单聊网络层
4. 单聊流式状态机
5. 单聊 UI
6. 单聊附件
7. 单聊兼容增强
8. 群聊数据层
9. 群聊状态机
10. 群聊 UI
11. 高级模块筛选迁移

这个顺序的重点是：

- 先做状态和契约
- 再做页面
- 先做单聊
- 再做群聊

不是相反。

---

## 9. 开工门槛

在真正开始写 Android 代码之前，至少要满足这些门槛：

1. 当前文档集已足够让人不看桌面代码也知道要兼容什么
2. 已经接受首版不做桌面全量复刻
3. 已经接受群聊后置
4. 已经接受很多逻辑短期内仍在客户端

如果这四条没有共识，直接开工只会反复返工。

---

## 10. 当前结论

Android 首版最合理的定义是：

- 一个能稳定连接现有 `VCPToolBox` 的 `VCPChat` 单聊客户端

不是：

- 一个第一天就试图覆盖桌面端全部能力的移动端巨系统

先把这个范围钉死，后面的实现节奏才会正常。
